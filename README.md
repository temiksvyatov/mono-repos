## Описание проекта

Этот проект представляет собой монорепозиторий для базовых Docker-образов, используемых в различных микросервисах. Он включает в себя шаблоны, конфигурации и скрипты для генерации и сборки образов.

## Структура проекта

- **common**: Общие файлы и шаблоны.

  - **files**: Общие конфигурационные файлы (например, `nginx.conf`, `pip.conf`).
  - **templates**: Общие шаблоны Dockerfile (например, `Dockerfile.common.j2`).

- **images**: Каталоги для конкретных образов.

  - **alpine**: Конфигурация для Alpine-образов.
  - **golang**: Конфигурация для Go-образов.
 - **java**: Конфигурация для Java-образов.
  - **jre**: Конфигурация для JRE-образов.
  - **nginx**: Конфигурация для Nginx-образов.
  - **node**: Конфигурация для Node.js-образов.
  - **python**: Конфигурация для Python-образов.

  Для каждого образа используется следующая структура:

  - `images/<image>/config.yaml` — базовый конфиг образа.
  - `images/<image>/Dockerfile.j2` — шаблон Dockerfile по умолчанию для образа.
  - `images/<image>/<version>/config.yaml` — (опционально) per-version override конфига.
  - `images/<image>/<version>/Dockerfile.j2` — (опционально) per-version шаблон Dockerfile.

  Если per-version файлы существуют, они переопределяют базовые `config.yaml`/`Dockerfile.j2` для конкретной версии.

- **jenkins**: Скрипты и конфигурации для Jenkins.

  - **builder**: Скрипты для сборки образов.
  - **dockerfile**: Скрипты для генерации Dockerfile.
  - **report**: Скрипты для генерации отчетов.
  - **validation**: Скрипты для валидации конфигураций.

- **generate_dockerfile.py**: Скрипт для генерации Dockerfile на основе шаблонов и конфигураций.

- **versions.yaml**: Файл с версиями и базовыми образами для каждого типа образов.
  - Для каждой категории/подкатегории (alpine, golang, node, python, nginx, java/maven, java/gradle, jre) задаются:
    - `versions`: список версий с обязательными полями `version` и `base_image`.
    - `image_tag_format`: строка формата основного тега образа, содержащая плейсхолдер `{version}` (например, `docker-mf-middle-dev-local.nexign.com/microservices/infra/build/node/docker-node{version}-alpine:latest`).
    - (опционально) per-version поля, например:
      - `image_tag_format` — переопределение формата основного тега только для конкретной версии.
      - `extra_tags` — список дополнительных форматов тегов для одной и той же версии образа (multi-tag), также с плейсхолдером `{version}`.
      - `multi_registry: true|false` — флаг, что версия (или вся категория/подкатегория) должна реплицироваться в дополнительные реестры.
  - Допускаются иерархические overrides:
    - корневая категория (`java`) может задать `multi_registry: true` по умолчанию,
    - подкатегория (`java.maven`) может переопределить это значение,
    - конкретная версия (`java.maven` 17) имеет наивысший приоритет.
  - Jenkins-пайплайн использует эти значения напрямую через функции `getImageTags`/`getImageTag` в `jenkins/builder/ImageBuilder.groovy` и через `getTargetTags` в `jenkins/pusher/ImagePusher.groovy`; YAML является единственным источником правды для схемы тегов, дополнительных тегов и multi-registry-флагов.

## Конфигурация

### Общие конфигурации

- **nginx.conf**: Конфигурация Nginx.
- **pip.conf**: Конфигурация pip для Python-образов.

### Конфигурации для конкретных образов

Каждая директория в `images` содержит свои конфигурационные файлы и шаблоны Dockerfile. Например:

- Для Go-образов используется `images/golang/config.yaml` + `images/golang/Dockerfile.j2`.
- Для Python-образов используется `images/python/config.yaml` + `images/python/Dockerfile.j2`, а также могут быть per-version overrides, например `images/python/314/Dockerfile.j2`.

## Использование

### Генерация Dockerfile

Для генерации Dockerfile используется скрипт `generate_dockerfile.py`. Он принимает имя образа и данные версии, а затем:

- Собирает финальный конфиг как merge из:
  - `common/config.yaml` (`default`),
  - `images/<image>/config.yaml`,
  - `images/<image>/<version>/config.yaml` (если существует),
  - записи версии из `versions.yaml`.
- Выбирает шаблон в следующем порядке:
  - `images/<image>/<version>/Dockerfile.j2` (если существует),
  - `images/<image>/Dockerfile.j2`,
  - `common/templates/Dockerfile.common.j2` (fallback).
- Генерирует Dockerfile в `generated/<image>/<version>/Dockerfile`.

### Сборка образов

Сборка образов выполняется с помощью Jenkins. В `Jenkinsfile` определены этапы для валидации, генерации Dockerfile, сборки образов, smoke-тестов, публикации образов и (опционально) дополнительных проверок.

## Сборка образов

Сборка образов выполняется в Jenkins с использованием параметров, таких как режим сборки (параллельный или последовательный), список образов для сборки и URL реестра Docker.

### Пример параметров

- **BUILD_MODE**: Режим сборки (parallel или sequential).
- **IMAGES_TO_BUILD**: Список образов для сборки (all или через запятую, например, alpine,golang,java/maven).
- **REGISTRY_URL**: URL реестра Docker (используется для аутентификации при сборке/пуше).
- **REGISTRY_NAMESPACE**: Базовый namespace в реестре (используется при авторизации и для обратной совместимости).
- **REGISTRY_CREDENTIALS**: ID учетных данных для реестра Docker.
- **BUILDER_IMAGE**: Образ для сборки Dockerfile.
- **MAX_PARALLEL_THREADS**: Максимальное количество параллельных потоков сборки.
- **TARGET_PLATFORMS**: (опционально) список целевых платформ для будущих multi-arch сборок (например, `linux/amd64,linux/arm64`).
- **GENERATE_AND_SEND_REPORT**: Включение генерации и отправки итогового отчета.
- **ENABLE_DOCKER_LINT**: (опционально) запуск `hadolint` для проверок сгенерированных Dockerfile.
- **ENABLE_DOCKER_SCAN**: (опционально) запуск `trivy` для сканирования собранных образов.
- **EXTRA_REGISTRIES**: (опционально) список дополнительных реестров (через запятую), в которые нужно реплицировать образы после пуша в основной реестр.

## Отчеты

После выполнения сборки Jenkins генерирует отчет в формате HTML, который содержит информацию о статусе сборки каждого образа.

## Валидация

Перед сборкой образов выполняется валидация конфигураций с помощью скриптов в `jenkins/validation`. Это включает:

- Проверку наличия директорий и базовых файлов `config.yaml`/`Dockerfile.j2` для всех образов.
- Проверку того, что каждая версия из `versions.yaml` имеет `base_image` и `version`.
- Проверку per-version overrides: если существует директория `images/<image>/<version>/`, в ней должен быть хотя бы один из файлов `config.yaml` или `Dockerfile.j2`.

## Multi-registry репликация образов

После успешной сборки и smoke-тестов стадия `Push Images to Registry`:

- Пушит каждый образ в основной реестр в соответствии с основным `image_tag_format` из `versions.yaml` (через обычный `docker push`).
- Для этого же образа формируется полный список тегов:
  - основной тег;
  - дополнительные теги из `extra_tags` на уровне категории/подкатегории/версии (multi-tag).
- Если задан параметр `EXTRA_REGISTRIES` и итоговый флаг `multi_registry` для данной версии равен `true` (с учётом overrides на уровне категории/подкатегории/версии), то:
  - каждый тег (основной и дополнительные) реплицируется во все указанные реестры.

Формат параметра `EXTRA_REGISTRIES`:

- Строка вида: `registry1.example.com,registry2.example.com`.
- Для каждого успешно запушенного базового тега `<registry>/<path>:<tag>` формируются дополнительные теги:
  - `<registry1.example.com>/<path>:<tag>`
  - `<registry2.example.com>/<path>:<tag>`

Репликация выполняется командой:

- `docker buildx imagetools create -t <extraTag> <baseTag>`

Таким образом:

- **Имена тегов и реестры всегда соответствуют фактическому месту хранения образа**, где каждый дополнительный тег и реестр явно отражены в конфигурации.
- Используется уровень манифестов (`imagetools`), что хорошо сочетается с multi-arch сборками.

## Точная пересборка по изменённым версиям

- На стадии `Initial Validation`:
  - `utils.getChangedFiles` определяет список изменённых файлов (`git diff --name-only`).
  - `utils.getChangedImages` вычисляет, какие образы затронуты (например, `node`, `java/maven`).
  - `utils.getChangedVersions` дополнительно вычисляет, какие конкретные версии внутри образов были затронуты (например, `node` → `['22']`, если изменён `images/node/22/config.yaml`).
  - Эта информация сериализуется в `env.CHANGED_VERSIONS` и используется на последующих стадиях.
- На стадии генерации Dockerfile:
  - `generate_dockerfile.py` читает `CHANGED_VERSIONS` и, если для данного образа указаны конкретные версии, генерирует Dockerfile только для них.
- На стадии сборки:
  - `ImageBuilder.groovy` читает `CHANGED_VERSIONS` и ограничивает список версий, которые реально попадают в план сборки.

Таким образом, изменения per-version файлов (`images/<image>/<version>/...`) приводят к пересборке только соответствующих версий, а изменения базовых файлов образа (`images/<image>/...`) затрагивают все версии этого образа.
